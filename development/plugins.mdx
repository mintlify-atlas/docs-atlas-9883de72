# Building Plugins

LobeHub supports an extensible plugin system that allows agents to interact with external tools, APIs, and services. This guide covers plugin development from concept to deployment.

## Plugin Types

LobeHub supports two main plugin types:

### 1. Chat Plugins (Legacy)

**Function Calling plugins** - Traditional OpenAI-style function calling

Location: External repositories, indexed via [@lobehub/chat-plugins-gateway](https://github.com/lobehub/chat-plugins-gateway)

### 2. MCP Servers (Modern)

**Model Context Protocol** - Standardized protocol for tool integration

Location: Built-in (`packages/builtin-tools/`) or external MCP servers

**MCP is recommended** for new plugins due to:
- Better standardization
- Broader ecosystem
- Enhanced security
- Easier development

## MCP Server Development

### Quick Start

MCP servers expose tools through a standardized protocol. Here's a minimal example:

```typescript
import { Server } from '@modelcontextprotocol/sdk/server/index.js';
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js';

const server = new Server(
  {
    name: 'my-tool-server',
    version: '1.0.0',
  },
  {
    capabilities: {
      tools: {},
    },
  }
);

// Define a tool
server.setRequestHandler('tools/list', async () => ({
  tools: [
    {
      name: 'get_weather',
      description: 'Get weather for a location',
      inputSchema: {
        type: 'object',
        properties: {
          location: { type: 'string', description: 'City name' },
        },
        required: ['location'],
      },
    },
  ],
}));

// Handle tool calls
server.setRequestHandler('tools/call', async (request) => {
  const { name, arguments: args } = request.params;
  
  if (name === 'get_weather') {
    const weather = await fetchWeather(args.location);
    return {
      content: [
        {
          type: 'text',
          text: `Weather in ${args.location}: ${weather}`,
        },
      ],
    };
  }
  
  throw new Error(`Unknown tool: ${name}`);
});

// Start server
const transport = new StdioServerTransport();
await server.connect(transport);
```

### Implement MCP Server

```typescript
// src/index.ts
import { Server } from '@modelcontextprotocol/sdk/server/index.js';
import { tools } from './tools';
import { handleToolCall } from './handlers';

export function createMyFeatureServer() {
  const server = new Server(
    { name: 'myfeature', version: '1.0.0' },
    { capabilities: { tools: {} } }
  );

  server.setRequestHandler('tools/list', async () => ({ tools }));
  server.setRequestHandler('tools/call', handleToolCall);

  return server;
}
```

### Define Tools

```typescript
// src/tools.ts
import { Tool } from '@modelcontextprotocol/sdk/types.js';

export const tools: Tool[] = [
  {
    name: 'myfeature_action',
    description: 'Perform a specific action',
    inputSchema: {
      type: 'object',
      properties: {
        input: {
          type: 'string',
          description: 'Input parameter',
        },
        options: {
          type: 'object',
          properties: {
            flag: { type: 'boolean' },
          },
        },
      },
      required: ['input'],
    },
  },
];
```

### Implement Handlers

```typescript
// src/handlers.ts
import { CallToolRequest, CallToolResult } from '@modelcontextprotocol/sdk/types.js';

export async function handleToolCall(
  request: CallToolRequest
): Promise<CallToolResult> {
  const { name, arguments: args } = request.params;

  switch (name) {
    case 'myfeature_action': {
      const result = await performAction(args.input, args.options);
      return {
        content: [
          {
            type: 'text',
            text: JSON.stringify(result),
          },
        ],
      };
    }
    default:
      throw new Error(`Unknown tool: ${name}`);
  }
}

async function performAction(input: string, options?: any) {
  // Your implementation
  return { success: true, data: `Processed: ${input}` };
}
```

## Best Practices

### Tool Design

- **Single responsibility** - One tool, one task
- **Clear descriptions** - Help LLM understand when to use tool
- **Structured output** - Return JSON for easy parsing
- **Error handling** - Graceful failures with helpful messages

### Schema Design

```typescript
// ✅ Good - Clear, specific schema
{
  type: 'object',
  properties: {
    city: {
      type: 'string',
      description: 'City name (e.g., "San Francisco")',
      examples: ['New York', 'Tokyo'],
    },
    units: {
      type: 'string',
      enum: ['celsius', 'fahrenheit'],
      description: 'Temperature units',
      default: 'celsius',
    },
  },
  required: ['city'],
}

// ❌ Bad - Vague, unclear
{
  type: 'object',
  properties: {
    data: { type: 'string' }, // What data?
  },
}
```

### Security

- **Validate inputs** - Never trust LLM-provided arguments
- **Rate limiting** - Prevent abuse
- **Authentication** - Use API keys for external services
- **Sandboxing** - Isolate code execution

```typescript
import { z } from 'zod';

const ArgsSchema = z.object({
  location: z.string().min(1).max(100),
  units: z.enum(['celsius', 'fahrenheit']).optional(),
});

export async function handleToolCall(request) {
  // Validate
  const args = ArgsSchema.parse(request.params.arguments);
  
  // Process safely
  const result = await getWeather(args.location, args.units);
  return { content: [{ type: 'text', text: result }] };
}
```

### Performance

- **Async operations** - Don't block the event loop
- **Caching** - Cache external API responses
- **Timeouts** - Set reasonable timeout limits
- **Streaming** - For long operations, stream results
