## What is MCP?

The **Model Context Protocol (MCP)** is an open standard that enables seamless, secure connections between AI agents and external tools, data sources, and services.

Think of MCP as a universal adapter that lets your agents interact with:

- Databases (PostgreSQL, MySQL, MongoDB)
- APIs (REST, GraphQL, custom endpoints)
- File systems (local, cloud storage, Git repositories)
- Development tools (IDEs, build systems, package managers)
- Business applications (CRM, project management, analytics)
- And virtually any other external service

<Note>
  MCP breaks down the barriers between AI and the digital ecosystem, transforming agents from isolated chatbots into connected teammates that can actually do work.
</Note>

## Why MCP matters

### Before MCP

Traditional AI assistants are limited:

- **Isolated**: Can only chat, can't interact with your tools
- **Static**: No access to real-time data or external services
- **Manual**: You must copy/paste data between systems
- **Fragmented**: Each tool needs custom integration

## How MCP works

### Architecture overview

<Steps>
  <Step title="MCP Server">
    An MCP server is a program that exposes tools and resources:
    - Runs as a separate process
    - Implements the MCP protocol
    - Provides tools (functions agents can call)
    - Offers resources (data agents can access)
  </Step>
  <Step title="MCP Client (LobeHub)">
    LobeHub acts as an MCP client:
    - Discovers available MCP servers
    - Manages connections
    - Routes agent requests to appropriate servers
    - Handles authentication and permissions
  </Step>
  <Step title="Agent interaction">
    When an agent needs external capabilities:
    - Agent determines which MCP tool to use
    - LobeHub sends request to MCP server
    - Server executes and returns results
    - Agent incorporates results into response
  </Step>
</Steps>

### Connection types

MCP supports two connection methods:

<Accordion title="STDIO (Standard Input/Output)">
  **How it works**: MCP server runs as a local process, communicates via stdin/stdout
  
  **Use cases**:
  - Local tools and scripts
  - Desktop applications
  - File system access
  - Development tools
  
  **Example**:
  ```json
  {
    "type": "stdio",
    "command": "npx",
    "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/workspace"]
  }
  ```
</Accordion>

<Accordion title="HTTP/HTTPS">
  **How it works**: MCP server runs as a web service, communicates over HTTP
  
  **Use cases**:
  - Cloud services
  - Remote APIs
  - Multi-user tools
  - Web applications
  
  **Example**:
  ```json
  {
    "type": "http",
    "url": "https://api.example.com/mcp",
    "auth": {
      "type": "bearer",
      "token": "your-api-token"
    }
  }
  ```
</Accordion>

### Authentication

MCP supports multiple authentication methods:

<AccordionGroup>
  <Accordion title="None (public access)">
    No authentication required—suitable for public data sources.
  </Accordion>
  <Accordion title="Bearer token">
    Simple token-based authentication:
    ```json
    {
      "auth": {
        "type": "bearer",
        "token": "your-api-token"
      }
    }
    ```
  </Accordion>
  <Accordion title="OAuth 2.0">
    Full OAuth flow for services like Google, GitHub, Microsoft:
    ```json
    {
      "auth": {
        "type": "oauth2",
        "accessToken": "obtained-from-oauth-flow"
      }
    }
    ```
  </Accordion>
  <Accordion title="Custom headers">
    API key or custom authentication:
    ```json
    {
      "headers": {
        "X-API-Key": "your-api-key",
        "X-Custom-Auth": "custom-value"
      }
    }
    ```
  </Accordion>
</AccordionGroup>

## Using MCP with agents

### Automatic tool selection

Agents automatically use MCP tools when relevant:

<CodeGroup>
```plaintext Database Query
User: "How many users signed up last month?"

Agent: *Detects need for database access*
        *Uses PostgreSQL MCP to execute query*
        
        SELECT COUNT(*) FROM users 
        WHERE created_at >= DATE_TRUNC('month', CURRENT_DATE - INTERVAL '1 month')
        AND created_at < DATE_TRUNC('month', CURRENT_DATE)
        
        "247 users signed up last month."
```

User: "Create a new Python file called utils.py with a function to validate emails"

```plaintext External API

```
</CodeGroup>

## Creating MCP servers

### When to create custom MCP servers

Build your own MCP server to:

- Connect to proprietary internal tools
- Integrate custom APIs or databases
- Expose domain-specific functionality
- Create reusable tools for your team

### MCP server basics

An MCP server provides:

<Accordion title="Tools (functions)">
  Callable functions agents can execute:
  
  ```typescript
  {
    name: "create_ticket",
    description: "Create a new support ticket",
    parameters: {
      title: { type: "string", required: true },
      description: { type: "string", required: true },
      priority: { type: "string", enum: ["low", "medium", "high"] }
    }
  }
  ```
</Accordion>

<Accordion title="Resources (data)">
  Data sources agents can read:
  
  ```typescript
  {
    uri: "database://tickets/recent",
    name: "Recent Tickets",
    description: "Tickets from the last 7 days",
    mimeType: "application/json"
  }
  ```
</Accordion>

<Accordion title="Prompts (templates)">
  Pre-defined prompt templates:
  
  ```typescript
  {
    name: "analyze_ticket",
    description: "Analyze a support ticket",
    arguments: [{ name: "ticket_id", required: true }]
  }
  ```
</Accordion>

## Advanced features

### Environment variables

Use environment variables for sensitive configuration:

<CodeGroup>
```json Connection Config
{
  "type": "stdio",
  "command": "python",
  "args": ["mcp_server.py"],
  "env": {
    "API_KEY": "${MCP_API_KEY}",
    "DATABASE_URL": "${MCP_DATABASE_URL}"
  }
}
```

```bash Environment Setup
# In your .env file
MCP_API_KEY=your-secret-key
MCP_DATABASE_URL=postgresql://localhost/db
```
</CodeGroup>

### Resource subscriptions

Agents can subscribe to resource changes:

```typescript
// Agent subscribes to new tickets
subscribe("database://tickets/new")

// When new ticket is created:
→ Agent is notified automatically
→ Can process immediately
```

### Sampling and prompts

MCP servers can provide:

- **Sampling**: Request agent to process specific prompts
- **Prompt templates**: Pre-configured prompts for common tasks

## Security and permissions

### Permission model

<AccordionGroup>
  <Accordion title="Connection-level permissions">
    Control whether an MCP server can connect:
    - Approved servers only
    - Review before first connection
    - Revoke access anytime
  </Accordion>
  <Accordion title="Tool-level permissions">
    Control which tools agents can use:
    - Enable/disable specific tools
    - Read-only vs. read-write access
    - Require approval for destructive operations
  </Accordion>
  <Accordion title="Resource-level permissions">
    Control access to resources:
    - Limit to specific directories/databases
    - Restrict sensitive data
    - Audit access logs
  </Accordion>
</AccordionGroup>

### Best practices

<CardGroup cols={2}>
  <Card title="Principle of least privilege" icon="key">
    Grant minimum permissions needed. Only enable tools agents actually use.
  </Card>
  <Card title="Use environment variables" icon="lock">
    Never hardcode secrets. Store in environment variables or secure vaults.
  </Card>
  <Card title="Review before approving" icon="eye">
    Carefully review MCP server capabilities before installation.
  </Card>
  <Card title="Monitor usage" icon="chart-line">
    Track which tools are used and how often. Audit for unexpected activity.
  </Card>
</CardGroup>

### Security checklist

<Steps>
  <Step title="Verify source">
    Only install MCP plugins from trusted sources.
  </Step>
  <Step title="Review permissions">
    Check what access the MCP server requests.
  </Step>
  <Step title="Test in isolation">
    Test new MCP servers with non-production data first.
  </Step>
  <Step title="Use read-only mode">
    Start with read-only access, upgrade only if needed.
  </Step>
  <Step title="Rotate credentials">
    Regularly update API keys and tokens.
  </Step>
</Steps>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Connection failed">
    **STDIO servers**:
    - Verify command exists: `which npx` or `which python`
    - Check file paths are absolute
    - Ensure dependencies are installed
    
    **HTTP servers**:
    - Verify URL is reachable
    - Check authentication credentials
    - Review CORS settings
  </Accordion>
  <Accordion title="Tools not appearing">
    **Solutions**:
    - Reconnect to MCP server
    - Check server logs for errors
    - Verify server implements MCP protocol correctly
    - Review tool manifest
  </Accordion>
  <Accordion title="Authentication errors">
    **Solutions**:
    - Verify API key is correct
    - Check token hasn't expired
    - Review authentication type matches server requirements
    - Test credentials manually
  </Accordion>
  <Accordion title="Permission denied">
    **Solutions**:
    - Check file/database permissions
    - Verify agent has appropriate access level
    - Review MCP server permission settings
    - Ensure environment variables are set
  </Accordion>
</AccordionGroup>
